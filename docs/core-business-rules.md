# ERP Core Business Rules

작성일: 2026-07-04

이 문서는 결제 요청, 승인, 지급 흐름의 1차 업무 규칙 기준이다. 프론트 mock 기능, API 명세, DB 설계, 감사 로그 설계는 이 기준을 우선 따른다.

## 결제 요청 상태 전이

| 현재 상태 | 허용 다음 상태 | 허용 액션 | 수정 가능 범위 |
| --- | --- | --- | --- |
| 임시 저장 | 제출 | 임시 저장, 제출 | 전체 필드 |
| 제출 | 승인 대기 | 시스템 접수 | 요청자 수정 불가 |
| 승인 대기 | 승인 진행 중, 반려 | 승인자 검토 시작, 반려 | 요청자 수정 불가 |
| 승인 진행 중 | 승인 완료, 반려, 보류 | 승인, 반려, 보류 | 승인자 메모만 가능 |
| 보류 | 승인 진행 중, 반려 | 재검토, 반려 | 승인자 메모만 가능 |
| 반려 | 임시 저장, 제출 | 보완, 재상신 | 요청자 보완 가능 |
| 승인 완료 | 없음 | 조회만 가능 | 수정 불가 |

## 승인 상태 전이

| 현재 상태 | 승인 | 반려 | 보류 | 중복 처리 |
| --- | --- | --- | --- | --- |
| 승인 대기 | 가능 | 가능 | 가능 | rowVersion 검증 |
| 승인 진행 중 | 가능 | 가능 | 가능 | rowVersion 검증 |
| 보류 | 재개 후 가능 | 가능 | 유지 | 보류 사유 필수 |
| 반려 | 불가 | 불가 | 불가 | 재상신으로 새 흐름 생성 |
| 승인 완료 | 불가 | 불가 | 불가 | 동일 승인 차단 |

## 지급 상태 전이

| 현재 상태 | 지급 실행 | 보류 | 오류 처리 | 수정 가능 범위 |
| --- | --- | --- | --- | --- |
| 지급 예정 | 계좌 확인 완료 시 가능 | 가능 | 가능 | 지급 예정일, 메모 |
| 오늘 지급 | 계좌 확인 완료 시 가능 | 가능 | 가능 | 지급 메모 |
| 보류 | 재개 후 가능 | 유지 | 가능 | 보류 사유, 지급 예정일 |
| 오류 | 재검증 후 예정 전환 | 가능 | 유지 | 오류 메모, 계좌 재확인 |
| 지급 완료 | 불가 | 불가 | 불가 | 조회만 가능 |

## 예산 상태 전이

| 현재 상태 | 다음 상태 | 조건 | 허용 액션 |
| --- | --- | --- | --- |
| 정상 | 주의 | 사용률 80% 이상 | 조회, 조정, 요청 연결 |
| 주의 | 정상 | 예산 조정 또는 사용률 80% 미만 회복 | 조회, 조정, 요청 연결 |
| 주의 | 초과 | 사용률 100% 이상 | 조회, 조정, 추가 승인 요청 |
| 초과 | 주의, 정상 | 예산 증액 또는 요청 취소로 사용률 회복 | 조회, 조정, 예외 승인 |
| 정상, 주의, 초과 | 마감 | 회계 기간 마감 | 조회, 이력 확인 |
| 마감 | 없음 | 마감 해제는 관리자 보정 절차 필요 | 조회만 가능 |

예산 금액 계산은 `allocatedAmount - usedAmount`를 기본 잔액으로 보고, 지급 완료 시점에 `usedAmount`를 확정 반영한다. 지급 예정 단계에서는 예약 사용액으로 별도 집계한다.

## 거래처 상태 전이

| 현재 상태 | 다음 상태 | 조건 | 허용 액션 |
| --- | --- | --- | --- |
| 활성 | 비활성 | 더 이상 신규 요청에 사용하지 않음 | 조회, 이력 확인 |
| 활성 | 차단 | 계좌 불일치, 법무/컴플라이언스 사유 | 조회, 계좌 재확인, 지급 차단 |
| 비활성 | 활성 | 재사용 승인 | 조회, 수정, 신규 요청 허용 |
| 차단 | 활성 | 계좌 재확인 및 관리자 승인 | 조회, 수정, 신규 요청 허용 |
| 차단 | 비활성 | 거래 중단 확정 | 조회, 이력 확인 |

비활성 또는 차단 거래처는 신규 결제 요청 선택 대상에서 제외한다. 이미 승인 또는 지급 완료된 과거 건은 원본 거래처 정보를 유지한다.

## 권한 매트릭스

| 역할 | 주요 권한 | 제한 |
| --- | --- | --- |
| 요청자 | 결제 요청 생성, 임시 저장, 제출, 반려 건 보완 | 본인 요청만 수정 가능 |
| 승인자 | 승인 대기 건 조회, 승인, 반려, 보류 | 지정 결재선의 현재 단계만 처리 가능 |
| 재무팀 | 지급 예정 조회, 지급 실행, 지급 보류, 거래처/예산 조회 | 승인 완료 건만 지급 가능 |
| 관리자 | 결재 정책, 권한, 사용자, 시스템 설정 관리 | 감사 로그 삭제 불가 |
| 외부 감사 | 감사 로그와 보고서 조회 | 업무 액션 불가 |

## 메뉴 접근 권한

| 메뉴 | 필요 권한 |
| --- | --- |
| 대시보드 | `dashboard:read` |
| 결제 요청 | `payment_request:read_own` 또는 `payment_request:read_all` |
| 승인 관리 | `approval:read_assigned` |
| 지급 관리 | `disbursement:read` |
| 예산 관리 | `budget:read` |
| 거래처 관리 | `vendor:read` |
| 보고서 | `report:read` |
| 시스템 설정 | `system:manage` |
| 즐겨찾기 | `favorite:read` |

관리자는 `*` 권한으로 모든 메뉴에 접근할 수 있다.

## 데이터 조회 범위

| 역할 | 기본 조회 범위 |
| --- | --- |
| 요청자 | 본인이 생성한 결제 요청 |
| 승인자 | 본인에게 배정된 승인 단계의 결제 요청 |
| 재무팀 | 전체 결제 요청, 지급, 거래처, 예산 |
| 관리자 | 전체 데이터 |
| 외부 감사 | 감사 로그와 보고서 조회 전용 |

서버 API는 세션 사용자 기준으로 조회 범위를 다시 계산한다. 프론트 메뉴 제한은 편의 기능이며, 실제 권한 검증은 백엔드에서 반복 검증한다.

## 결재선 자동 생성 규칙

| 금액 구간 | 결재 단계 | 필수 승인자 |
| --- | --- | --- |
| 1,000,000원 이하 | 요청자 → 1차 승인자 | 1명 |
| 1,000,001원 ~ 5,000,000원 | 요청자 → 1차 승인자 → 부서장 | 2명 |
| 5,000,001원 ~ 20,000,000원 | 요청자 → 1차 승인자 → 부서장 → 재무팀 | 3명 |
| 20,000,001원 이상 | 요청자 → 부서장 → 재무팀 → 최종 승인자 | 4명 |

예산 초과, 특정 거래처 예외, 관리자 지정 예외가 있으면 재무팀 승인 단계를 추가한다.

## 예산 초과 처리 규칙

- 사용률 80% 이상은 주의 상태로 표시한다.
- 사용률 100% 이상은 초과 상태로 표시하고 기본 제출을 제한한다.
- 5,000,000원 이상 요청이거나 예산 초과 위험이 있으면 재무팀 추가 승인을 요구한다.
- 초과 승인 예외는 사유와 승인자를 감사 로그에 남긴다.

## 첨부파일 처리 규칙

- 허용 확장자는 PDF, JPG, JPEG, PNG, XLSX로 제한한다.
- 파일 1개당 최대 용량은 10MB로 제한한다.
- 임시 저장 또는 반려 상태에서 요청 수정 권한이 있는 사용자만 증빙 파일을 추가하거나 삭제할 수 있다.
- 제출 이후 승인 진행 중인 건과 승인 완료 건은 증빙 파일 변경을 차단한다.
- 실패한 업로드는 파일명과 사유를 사용자에게 즉시 표시한다.

## 알림 처리 규칙

- 승인 요청, 반려, 보류, 승인 완료, 지급 예정, 지급 완료, 예산 초과, 결재 지연, 시스템 설정 변경은 알림 대상으로 본다.
- 알림은 사용자별로 저장하며 목록에서는 최신 30건을 우선 표시한다.
- 읽음 처리는 개별 알림과 전체 알림 모두 지원한다.
- 기본 보관 기간은 생성일 기준 90일이며, 보관 기간이 지난 알림은 목록 조회에서 제외한다.
- 알림의 연결 화면은 `linkPath`로 제공하고, 프론트는 알림 클릭 시 해당 메뉴로 이동한다.

## 삭제 가능 조건

| 대상 | 삭제 가능 조건 | 삭제 불가 조건 |
| --- | --- | --- |
| 결제 요청 | 임시 저장 상태이며 요청자 본인 또는 관리자 | 제출 이후, 승인/반려/보류/지급 연계 이후 |
| 첨부파일 | 결제 요청이 임시 저장 또는 반려이고 수정 권한이 있음 | 제출 이후 승인 진행 중, 승인 완료, 지급 연계 이후 |
| 거래처 | 지급/결제 요청 이력이 없는 경우 | 결제 요청, 지급, 감사 로그에 참조된 경우 |
| 예산 항목 | 사용액과 연결 요청이 모두 0인 경우 | 결제 요청 또는 지급 이력 존재 |
| 즐겨찾기 | 사용자 본인의 즐겨찾기 | 다른 사용자의 즐겨찾기 |
| 보고서 정의 | 소유자 또는 관리자, 실행 이력이 보관 정책을 위반하지 않는 경우 | 예약 발송 활성 또는 감사 대상 보고서 |

실데이터 참조가 있는 업무 객체는 물리 삭제보다 비활성화 또는 숨김 처리를 우선한다.

## 업무 마감 이후 처리 제한

- 회계 기간이 마감되면 해당 기간의 결제 요청 신규 제출, 예산 조정, 지급 예정일 변경을 차단한다.
- 마감 이후 반려 건 재상신은 다음 회계 기간으로 새 요청을 생성한다.
- 마감 이후 지급 오류 복구는 관리자 수동 복구 절차와 감사 로그 사유가 필요하다.
- 마감 해제는 관리자 2인 승인과 사유 기록을 요구한다.
- 마감된 기간의 보고서는 재생성 가능하지만 기존 보고서 artifact는 보존한다.

## 중복 처리 및 동시성 기준

- 모든 승인, 반려, 보류, 지급 실행 요청은 `rowVersion` 또는 `updatedAt` 기준으로 최신 상태를 검증한다.
- 승인 완료, 반려, 지급 완료처럼 최종 상태인 건은 동일 액션 재처리를 차단한다.
- 일괄 처리 중 일부 실패는 성공 건과 실패 건을 분리해 사용자에게 보여준다.
- 네트워크 실패 후 재시도는 같은 `idempotencyKey`를 사용한다.
- 상태 변경은 업무 감사 로그에 남기고, 권한 오류, 중복 처리 차단, 부분 실패, 서버 실패는 `security_events.requestId`로 화면 오류 메시지와 운영 로그를 대사한다.
