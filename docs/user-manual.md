# User Manual

작성일: 2026-07-06

이 문서는 결제 요청 승인 ERP를 실제 업무 화면 기준으로 사용하는 절차다. 화면의 버튼은 권한과 상태에 따라 비활성화될 수 있으며, 실패 메시지에 `requestId`가 표시되면 운영자에게 그대로 전달한다.

## 로그인과 기본 탐색

1. 로그인 화면에서 회사 이메일과 비밀번호를 입력한다.
   비밀번호가 만료되었다는 메시지가 표시되면 같은 화면에서 현재 비밀번호와 새 비밀번호를 입력해 변경한 뒤 다시 로그인한다.
2. 대시보드에서 승인 대기, 오늘 마감, 지급 예정, 예산 주의 항목을 확인한다.
3. 좌측 메뉴에서 결제 요청, 승인 관리, 지급 관리, 예산 관리, 거래처, 보고서, 시스템 설정, 즐겨찾기로 이동한다.
4. 공통 표의 상단 검색과 화면별 필터, 정렬, 페이지네이션을 사용해 목록을 좁힌다.
5. 목록 행을 선택하면 우측 상세 패널에서 상세 정보, 첨부 파일, 가능한 작업, 비활성 사유를 확인한다.
6. 새로고침 또는 재로그인 후에도 같은 업무 데이터가 보여야 한다. 데이터가 사라지면 저장 완료 메시지와 `requestId`를 운영자에게 전달한다.

## 결제 요청

1. 결제 요청 화면에서 새 요청을 열고 거래처, 부서, 예산 항목, 금액, 요청 사유를 입력한다.
2. 거래처가 없으면 거래처 추가 흐름으로 등록한 뒤 결제 요청 master data를 다시 불러와 선택한다.
3. 증빙 파일은 PDF, JPG, JPEG, PNG, XLSX만 업로드한다. 10MB를 넘거나 확장자와 Content-Type이 맞지 않으면 업로드가 차단된다.
4. 제출 전 예산 잔액, 예상 결재선, 첨부 업로드 완료 상태를 확인한다.
5. 임시 저장은 draft 상태를 저장하고, 제출은 결재 단계와 알림을 생성한다.
6. 제출 이후에는 입력과 첨부 변경이 제한되며, 서버 validation 실패는 필드 또는 화면 메시지로 표시된다.

## 승인 관리

1. 승인 관리 화면에서 본인에게 배정된 승인 대기 또는 승인 진행 중 건을 확인한다.
2. 요청 상세, 첨부 파일, 결재선 현황, 예산 확인 상태를 검토한다.
3. 승인, 반려, 보류 중 하나를 선택한다. 반려와 보류에는 사유 입력이 필요하다.
4. 이미 처리된 건, 본인 요청 건, 결재 순서가 맞지 않는 건은 서버에서 거절된다.
5. 처리 후 다음 처리 가능 건이 자동 선택되며, 알림 센터와 대시보드는 같은 원천 데이터로 갱신된다.

## 지급 관리

1. 지급 관리 화면에서 지급 예정, 오늘 지급, 지급 완료, 오류 상태를 확인한다.
2. 지급 실행 전 승인번호, 금액, 거래처, 계좌 확인 완료 상태를 확인한다.
3. 지급 실행은 2인 확인이 필요하며, 확인자와 실행자는 서로 달라야 한다.
4. 보류에는 사유가 필요하고, 오류 건은 계좌 재확인 후 재처리한다.
5. 은행 이체 파일은 승인 완료, 계좌 확인 완료, 활성 거래처 조건을 모두 통과한 건만 생성된다.
6. 은행 결과 대사 후 ERP 지급 상태, 감사 로그, 거래처 지급 이력이 함께 갱신된다.

## 예산 관리

1. 예산 관리 화면에서 부서, 예산 항목, 배정 예산, 사용액, 잔액, 사용률, 상태를 확인한다.
2. 예산 조정은 사유를 입력해야 하며, 금액에 따라 승인 필요 상태가 될 수 있다.
3. 마감된 예산 기간에서는 신규 결제 요청 제출, 예산 조정, 지급 예정일 변경, 오류 지급 복구가 차단된다.
4. 필터 결과와 조정 이력은 DB 예산 원장 기준으로 표시된다.

## 거래처 관리

1. 거래처 관리 화면에서 거래처명, 사업자번호, 담당자, 은행, 계좌 확인 상태, 세금계산서 정보를 확인한다.
2. 신규 등록과 수정 시 거래처명, 사업자번호, 담당자, 은행명, 계좌번호, 세금계산서 수신 이메일을 입력한다.
3. 사업자번호 중복, 계좌번호 형식 오류, 세금계산서 이메일 오류는 backend validation으로 차단된다.
4. 사업자등록증, 통장사본, 세금계산서 파일은 파일 저장소에 업로드되며 다운로드와 삭제 권한이 검증된다.
5. 비활성화 전 진행 중 결제 요청과 지급 예약 영향 범위를 확인한다.

## 보고서

1. 보고서 화면에서 유형과 기간을 선택한 뒤 보고서를 생성한다.
2. 보고서 목록의 CSV/PDF 다운로드는 서버가 저장된 ReportRun 기준으로 생성한다.
3. 예약 발송은 수신자, 주기, 시간, 형식, 활성 상태를 입력한다.
4. 생성, 수정, 삭제, 예약 변경은 감사 로그에 남고 중복 요청은 idempotencyKey 기준으로 처리된다.

## 시스템 설정

- 시스템 설정은 관리자 또는 권한이 있는 운영자만 사용한다.
- 일반 사용자는 본인 권한으로 접근 가능한 메뉴와 버튼만 볼 수 있다.
- 권한 변경 직후 메뉴와 API 권한이 달라질 수 있으므로, 접근이 거절되면 새로고침 후 다시 확인한다.

## 즐겨찾기

1. 즐겨찾기에서 자주 쓰는 메뉴와 저장 필터를 추가한다.
2. 바로가기는 대상 화면, 필터 조건, 이름, 공유 범위를 저장한다.
3. 순서 편집 후 사용자 저장을 실행하면 다른 브라우저에서도 같은 FavoriteItem 기준으로 동기화된다.
4. 삭제는 확인 후 수행하며, 가능한 경우 undo로 복구한다.

## 오류와 문의

- 오류 메시지에 `requestId`가 있으면 그대로 운영자에게 전달한다.
- 권한 부족, 중복 요청, 오래된 화면 데이터, 파일 검사 실패, rate limit 오류는 같은 `requestId`로 감사 로그 또는 보안 이벤트에서 조회된다.
- 결제 요청, 승인, 지급, 파일 업로드 같은 핵심 업무가 실패하면 화면명, 요청번호 또는 지급번호, 발생 시각, 수행 버튼, `requestId`를 함께 전달한다.
