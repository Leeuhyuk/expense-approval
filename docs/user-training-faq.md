# User Training And Support FAQ

작성일: 2026-07-08

이 문서는 파일럿/UAT와 go-live 전 사용자 교육, 운영 FAQ, 오류 신고 양식, requestId 수집 방법을 표준화한다. 실제 교육 완료 증빙은 `ROLE_UAT_EVIDENCE_PATH`, go-live handoff, final acceptance evidence에 연결한다.

## 교육 대상과 실습

| 역할 | 필수 실습 |
| --- | --- |
| 요청자 | 결제 요청 생성, 거래처 선택/신규 등록 요청, 증빙 업로드, 임시 저장, 제출, 반려 후 수정 |
| 승인자 | 승인 대기 목록 필터, 첨부 확인, 승인/반려/보류 사유 입력, 다음 건 자동 선택 확인 |
| 재무팀 | 지급 대상 조회, 계좌 확인, 지급 보류, 은행 이체 파일 dry-run, 보고서 다운로드 |
| 관리자 | 권한 그룹, 사용자 권한, 부서/승인 한도, 외부 연동 테스트, 운영 모드 확인 |
| 외부 감사 | 감사 로그 조회, 보고서 조회, requestId 검색, 원문 개인정보 미노출 확인 |

## FAQ

| 질문 | 답변 |
| --- | --- |
| 화면에 권한 부족이 표시됩니다 | 관리자에게 역할/권한 그룹 확인을 요청하고, 화면의 requestId를 함께 전달합니다. |
| 파일 업로드가 실패합니다 | 파일 확장자, 10MB 제한, 바이러스 검사 상태를 확인하고 같은 파일로 재시도합니다. 계속 실패하면 오류 신고 양식을 제출합니다. |
| 승인 버튼이 비활성화됩니다 | 본인 승인 금지, 결재 순서, 이미 처리된 건, rowVersion 충돌일 수 있습니다. 상세 메시지와 requestId를 확인합니다. |
| 지급 실행이 막힙니다 | 승인 완료, 계좌 확인, 2인 확인, 마감 기간, 중복 지급 차단 조건을 확인합니다. 재무 책임자 승인 없이 우회하지 않습니다. |
| 새로고침 후 데이터가 사라진 것처럼 보입니다 | 필터/정렬/기간 조건을 초기화하고, 다른 브라우저에서도 조회합니다. 계속 보이지 않으면 요청번호와 requestId를 제출합니다. |
| 오류가 발생했는데 무엇을 보내야 하나요 | 오류 문구, requestId, 화면명, 누른 버튼, 요청/승인/지급/거래처/보고서 번호, 파일명, 재현 시각을 보냅니다. |

## requestId 수집 방법

- API 오류 toast 또는 화면 메시지에 표시된 `requestId`를 그대로 복사한다.
- 운영자에게 문의할 때는 requestId와 수행 버튼을 함께 전달한다.
- 운영자는 requestId로 API 로그, `AuditLog`, `security_events`, 알림, 파일 접근 이벤트를 대사한다.
- requestId가 없으면 발생 시각, 사용자 역할, 화면명, 버튼명, 대상 번호를 대신 기록한다.

## 교육 완료 기준

- 역할별 필수 실습을 최소 1회 완료한다.
- 오류 신고 양식 예시 1건을 제출해 requestId 전달 방식을 확인한다.
- 교육 중 발견된 P0/P1은 go-live 전 해결하거나 책임자 예외 승인과 보완 기한을 남긴다.
- 교육 자료와 FAQ 변경 사항은 go-live handoff와 사용자 공지에 반영한다.
